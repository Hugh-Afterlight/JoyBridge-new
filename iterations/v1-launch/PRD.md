# JoyBridge · PRD · v1-launch

> 本文档是 JoyBridge 重构版的 First Flight 首版 PRD。根目录的 `BRIEF.md` 是长期纲领，本文件只定义本次从旧版 MVP 到重构版 v1 的目标、范围和验收标准。

## 项目背景

JoyBridge 已有一个能运行的原生 macOS MVP：使用 Swift、SwiftUI、AppKit、GameController.framework 和 CoreGraphics CGEvent，实现手柄按钮到键盘动作的映射，并已具备 Accessibility 权限检测、自定义映射保存、目标手柄锁定、全局暂停、运行检查和诊断信息复制等能力。

本次不是扩张功能，而是把一个“能用的个人工具”重构成“职责清楚、状态可信、后续可维护”的长期项目。原仓库 `Hugh-Afterlight/JoyBridge` 只作为只读参考，不在原仓库做任何修改；本轮所有规格、代码和提交都发生在新仓库 `Hugh-Afterlight/JoyBridge-Refactor`。

## 本次目标

### 产品目标

1. 用户打开 App 后，可以立刻知道 JoyBridge 当前是否可用，缺少什么条件，下一步应该点哪里。
2. 用户可以稳定完成“连接手柄 -> 授权 Accessibility -> 锁定目标手柄 -> 修改映射 -> 输出键盘动作”的核心流程。
3. 暂停、恢复、目标锁定、权限检测和最近按键展示都有一致、可解释的行为。
4. 朋友测试者遇到问题时，可以复制诊断信息，而不是只能口头描述“不能用”。
5. 保留未来公开发布路径，但本轮仍以本地朋友测试为目标，不做公证和 App Store。

### 工程目标

1. 明确核心链路：ControllerDevice -> ControllerInput -> ControllerButton -> KeyMapping -> MappingAction -> KeyboardOutput -> DiagnosticReport。
2. 拆清 Presentation、Application、Domain、Infrastructure、Persistence、Diagnostics 的职责。
3. 让 SwiftUI View 只负责展示和触发用户操作，不直接承载复杂系统 API 和业务规则。
4. 让 Mapping 成为核心领域模型，而不是 UI 表单数据。
5. 保留 MappingProfile、JSON store、Universal build、Developer ID release 的扩展点，但不在本轮做完整功能。

## 核心用户与关键场景

### 用户

- **作者本人**：需要长期稳定使用，后续继续扩展设备和映射能力。
- **本地朋友测试者**：需要能安装、授权、测试、反馈问题，不要求阅读完整开发 README。

### 场景 1：首次打开

用户打开 JoyBridge 后，主窗口第一屏必须回答：

- 当前是否可以输出键盘事件。
- 是否缺 Accessibility 权限。
- 是否检测到手柄。
- 是否已锁定目标手柄。
- 映射当前是启用还是暂停。

### 场景 2：连接并识别手柄

用户通过 macOS 蓝牙设置连接 Joy-Con、Switch Pro Controller 或兼容手柄后，JoyBridge 可以重新检测控制器，并展示当前设备、目标设备和最近按键。若目标手柄未连接，不应自动切换到其他手柄。

### 场景 3：锁定目标手柄

目标锁定是输入链路规则，不是普通 UI 状态。只有来自目标手柄的输入才允许进入映射执行流程。未锁定时可以测试，但界面必须建议用户锁定当前手柄。

### 场景 4：编辑映射

用户能看到每个 ControllerButton 的 Source、Action、State：

- Source：手柄按钮，例如 A、B、DPad Up。
- Action：键盘动作，例如 Space、Escape、Command + C。
- State：启用、禁用、默认值、无动作。

### 场景 5：暂停和恢复

暂停映射只停止键盘输出，不停止输入监听、最近按键展示和目标手柄状态维护。恢复后无需重启 App 即可继续输出。

### 场景 6：测试和诊断

诊断信息至少包含 App 版本、macOS 版本、App 路径、Accessibility 状态、当前手柄、目标手柄、最近按键、当前映射、暂停状态和输出结果。

## 单窗口信息架构

本轮采用单窗口分区，不做复杂侧边栏，不做多层设置页。

1. **顶部总状态**：App 名称、版本、总体运行状态、复制诊断入口。
2. **运行检查区**：权限、手柄、目标锁定、映射输出四项总览。
3. **权限区**：Accessibility 状态、当前 App 路径、打开系统设置、重新检测。
4. **手柄区**：当前手柄、可用手柄、目标手柄、最近按键、重新检测、锁定当前、清除锁定。
5. **映射区**：按键列表、启用状态、修饰键、键盘按键、预览。
6. **运行控制区**：暂停映射、启用映射，说明暂停后的行为。
7. **诊断区**：诊断摘要、复制按钮、朋友测试反馈提示。

## 菜单栏结构

- JoyBridge 状态。
- 暂停映射 / 启用映射。
- 显示 JoyBridge。
- 重新检测手柄。
- 重新检测权限或请求授权。
- 复制诊断信息。
- 退出 JoyBridge。

## 范围

### 本轮纳入

- 手柄到键盘映射。
- 单键、组合键、纯修饰键、无动作。
- Accessibility 权限检测。
- 手柄连接检测。
- 目标手柄锁定。
- 映射暂停和恢复。
- 映射持久化。
- 运行检查。
- 诊断信息。
- 单窗口页面重整。
- 模块重构。
- 基础测试，优先覆盖映射和状态规则。

### 本轮预留但不做 UI

- 多 Mapping Profile。
- JSON 导入导出。
- 更丰富设备诊断。
- Universal build。
- Developer ID 签名和 Apple 公证。

### 本轮不做

- 键盘作为输入设备。
- 鼠标控制、摇杆鼠标、摇杆滚动、Motion mouse。
- 左右 Joy-Con 合并。
- 按 App 自动切换配置。
- 云同步或 iCloud 同步。
- 登录项启动。
- 自动更新。
- App Store 打包。
- 插件系统。
- 复杂快捷键录制器。
- Electron、Tauri、Python、Node.js 重写。

## 目标衡量

### 产品体验指标

- 首次完成可用映射：朋友测试者 5 分钟内完成。
- 权限授权成功率：朋友测试中 >= 80%。
- 已声明支持设备识别成功率：>= 90%。
- 目标锁定正确率：100%，锁定后其他手柄不触发映射。
- 映射修改成功率：>= 95%，修改后无需重启即可生效。
- 暂停行为正确率：100%，暂停后最近按键更新但不输出。
- 诊断信息可用率：>= 90%，反馈中包含关键字段。

### 工程质量指标

- 核心映射逻辑有测试覆盖：单键、组合键、纯修饰键、无动作。
- 输入和输出解耦：Controller 模块不直接调用键盘输出。
- View 和业务解耦：SwiftUI View 不直接处理复杂系统 API。
- 状态来源清晰：每个运行状态只有一个可信来源。
- 持久化可替换：UserDefaults 可被未来 JSON store 替换。
- 诊断字段稳定：每次复制的信息结构一致。

## 主要风险

- GameController.framework 对单只 Joy-Con 的实体按键暴露可能不完整，这是系统能力限制，不应在 UI 中承诺“所有物理按键必定可用”。
- Accessibility 权限和 App 安装路径强相关，朋友测试时容易授权到错误路径。
- 当前 Xcode 项目可能没有测试 target，本轮添加测试时需要小心修改 `.xcodeproj`。
- 重构期间需要保证现有 MVP 能力不回退，尤其是暂停释放 modifier、目标锁定不自动切换、关闭窗口后菜单栏继续运行。
